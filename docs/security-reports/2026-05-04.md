## 🔴 CRITICAL (Immediate action required)

### C-1 — `pypa/gh-action-pypi-publish` pinned to mutable branch ref `release/v1`

**File:** `.github/workflows/publish.yml`  
**Impact:** Supply-chain compromise of the PyPI publish pipeline.

`publish.yml` references `pypa/gh-action-pypi-publish@release/v1`. `release/v1` is a **Git branch**, not a tag or a commit SHA. The action maintainer (or a compromised maintainer account) can push new code to that branch at any time; the next publish run silently executes the new code. This action runs in the job that holds the OIDC `id-token: write` permission and authenticates directly to PyPI. An attacker who controls the branch content could exfiltrate the OIDC token and publish an arbitrary wheel to PyPI under the `regstack` package name — reaching every downstream consumer on their next `pip install`.

All other actions (`actions/checkout@v4`, `astral-sh/setup-uv@v3`, `actions/upload-artifact@v4`, `actions/download-artifact@v4`) are pinned to version tags, which are also mutable but require deliberate re-pointing by the action author — lower risk, but still not SHA-pinned.

**Recommended fix:** Pin every `uses:` directive to a full commit SHA. For `pypa/gh-action-pypi-publish` the current commit behind `release/v1` can be resolved with `git ls-remote https://github.com/pypa/gh-action-pypi-publish refs/heads/release/v1`. Add the tag as a comment for human readability:

```yaml
uses: pypa/gh-action-pypi-publish@<SHA> # release/v1
```

### C-2 — OAuth sign-up ignores `config.allow_registration = False`

**File:** `src/regstack/routers/oauth.py:502–522`  
**Impact:** Security policy bypass — operators cannot disable new-account creation.

`_resolve_user` creates a brand-new `BaseUser` whenever no existing account matches the Google identity. There is no check against `rs.config.allow_registration`. An operator who sets `allow_registration = False` — to run a closed deployment, implement invite-only access, or freeze user creation after a breach — will still have new accounts silently created for any OAuth sign-in from a previously unseen Google address. The self-service `POST /register` endpoint correctly checks `config.allow_registration`, making the inconsistency a security bypass.

**Recommended fix:** Add a guard at the start of the "create new account" branch:

```python
if not rs.config.allow_registration:
    raise _LinkConflictError("registration_disabled")
```

Map `registration_disabled` to `?error=registration_disabled` in the callback error handler, and document it in the OAuth design.

## 🟠 WARNING (Address within 1 week)

### W-1 — Admin `DELETE /admin/users/{id}` does not cascade `oauth_identities`

**File:** `src/regstack/routers/admin.py:131–134`  
**Impact:** Orphan `oauth_identities` rows referencing a deleted `user_id`; subsequent OAuth sign-in from the same Google account may create a new user when it should be rejected, or fail with an unhandled lookup error depending on backend.

The self-service `DELETE /account` handler (`account.py:233`) correctly calls `rs.oauth_identities.delete_by_user_id(user.id)` before deleting the user. The admin path calls only `rs.users.delete(user.id)` and `rs.pending.delete_by_email(user.email)`, skipping the identity cascade. On Mongo (no foreign-key constraints) the orphan row survives indefinitely. On SQL (foreign key `user_id → users.id`) the delete may raise an integrity error on Postgres or silently do nothing on SQLite (depending on PRAGMA settings).

**Recommended fix:** Mirror the self-service deletion cascade in the admin route:

```python
await rs.oauth_identities.delete_by_user_id(user.id)
await rs.users.delete(user.id)
await rs.pending.delete_by_email(user.email)
```

### W-2 — `POST /phone/start` and `DELETE /phone` crash with HTTP 500 for OAuth-only users

**Files:** `src/regstack/routers/phone.py:65` and `phone.py:149`  
**Impact:** Unhandled 500 for authenticated OAuth-only users; 500 vs 400 reveals account type to an attacker who controls an OAuth-only session.

Both endpoints call `rs.password_hasher.verify(payload.current_password, user.hashed_password)` without first checking whether `user.hashed_password` is `None`. For users who signed up via OAuth and never set a password, `hashed_password` is `None`. `pwdlib` raises `TypeError: hash must be str or bytes` — the exception propagates as HTTP 500.

Compare `account.py:23–31`, which correctly calls `_require_password_set(user)` before any `verify()` call.

**Recommended fix:** Add `_require_password_set(user)` at the top of each handler (before the lockout check and `verify()` call).

### W-3 — `OAuthConfig.enforce_mfa_on_oauth_signin` is a dead config flag

**Files:** `src/regstack/config/schema.py:64`; absent from `src/regstack/routers/oauth.py`  
**Impact:** Users who have SMS 2FA enabled can bypass the second factor by signing in through OAuth. An operator who sets this flag believing it will enforce MFA on OAuth logins gets no enforcement.

The flag is defined and surfaced in the OAuth setup wizard (`wizard/oauth_google/cli.py`, `writer.py`) but is never read by the OAuth callback or exchange handler. The flag's presence in the wizard makes the security gap invisible to operators.

**Recommended fix:** Either implement the check (in `_sign_in_complete`, after resolving the user, check `enforce_mfa_on_oauth_signin` and redirect through the MFA flow if the user has a registered phone), or remove the flag and document that MFA enforcement on OAuth is out of scope.

### W-4 — `completion_ttl_seconds` is defined but never enforced; session JWT sits in state row for up to 5 minutes

**Files:** `src/regstack/config/schema.py:71`; `src/regstack/routers/oauth.py:139–149`  
**Impact:** Widened race window for state-ID theft. A session JWT is written to the `oauth_states` row at callback time and consumed by `POST /oauth/exchange`. The intended completion window is 30 seconds (`completion_ttl_seconds = 30`), but the exchange endpoint checks against `state.expires_at`, which uses the full `state_ttl_seconds` (default 300 seconds). The session JWT therefore sits in the database for up to 5 minutes after a successful callback. Anyone who observes the `state_id` (e.g., from browser history, server access logs, or a timing attack against the database) and can call `POST /oauth/exchange` before the legitimate SPA does can steal a valid session. The `consume()` atomicity prevents double-exchange, but the larger window increases exposure time.

**Recommended fix:** Store a separate `result_expires_at = now + completion_ttl_seconds` when writing the result token onto the state row, and validate against it in the exchange endpoint rather than the original `expires_at`.

### W-5 — `POST /login/mfa-confirm` has no per-account or per-IP rate limit

**File:** `src/regstack/routers/login.py:106–131`  
**Impact:** An attacker who captures a valid `mfa_pending_token` (10-minute JWT) can brute-force the 6-digit OTP at full network speed until the 5-attempt code-level lockout fires. With 10 minutes and no per-second throttle, the window is wide enough for automated tooling.

The `MfaCodeRepo.verify()` deletes the code row on the 5th wrong guess (`config.sms_code_max_attempts`), so brute-force is limited in practice — but there is no `LockoutService.check()` call, no per-IP middleware, and no `record_failure` on wrong guesses at the route level. Contrast with the primary login, which gates on `LockoutService` before password verification.

**Recommended fix:** Call `rs.lockout.check(user.email)` and `rs.lockout.record_failure(user.email)` in the MFA confirm handler, reusing the same sliding-window lockout as the primary login route.

### W-6 — No minimum length enforced on `jwt_secret`

**Files:** `src/regstack/auth/jwt.py:118`; `src/regstack/config/schema.py:168–175`  
**Impact:** A host that sets `REGSTACK_JWT_SECRET=x` (1 character) receives no rejection. The `_warn_empty_secret` validator only checks for an empty string. Short secrets drastically weaken HMAC-SHA256 security and would be trivially brute-forced.

**Recommended fix:** Add a validator on `RegStackConfig` (or in `JwtCodec.__init__`) that raises a `ValueError` if `len(jwt_secret.get_secret_value()) < 32`, with a message directing the operator to `regstack init` or `regstack doctor`.

### W-7 — `login_max_per_minute` / `login_max_per_hour` config fields are dead code

**File:** `src/regstack/config/schema.py:148–149`  
**Impact:** An operator who sets these fields, expecting route-level rate limiting, receives no enforcement. The misleading presence of the config creates a false sense of security.

No middleware, decorator, or route handler reads these fields. The comment says "reserved for future route-level rate limiting (slowapi-style)" but there is no corresponding roadmap item or TODO. Hosts cannot safely rely on these fields for DoS protection.

**Recommended fix:** Remove the fields until the feature is implemented, or add a startup warning when they are set to non-default values (indicating that enforcement is not yet in place).

### W-8 — `PyJWKClient` JWKS fetch blocks the asyncio event loop

**File:** `src/regstack/oauth/providers/google.py:82`  
**Impact:** A slow or unresponsive Google JWKS endpoint blocks the event loop for up to 30 seconds (the `urllib.request` default timeout), stalling all concurrent requests on that worker.

`PyJWKClient` uses `urllib.request.urlopen` synchronously. There is no `asyncio.to_thread` wrapper or `run_in_executor` call. The `exchange_code` HTTP call uses `httpx.AsyncClient(timeout=10.0)` (non-blocking), making the JWKS path the only synchronous I/O in the OAuth flow.

**Recommended fix:** Wrap the `PyJWKClient` fetch in `asyncio.to_thread`:

```python
signing_key = await asyncio.to_thread(
    self._jwks_client.get_signing_key_from_jwt, id_token
)
```

Also set an explicit timeout: `PyJWKClient(jwks_url, cache_keys=True, timeout=10)`.

### W-9 — `mfa_login_started` and `phone_setup_started` hooks expose raw plaintext OTP codes

**Files:** `src/regstack/routers/login.py:158`; `src/regstack/routers/phone.py:103`  
**Impact:** Any host hook handler that logs or forwards these events (monitoring, audit log, webhook) will capture live, usable OTP codes. These codes are valid for 5 minutes and give full MFA bypass capability.

`hooks.fire("mfa_login_started", user=user, code=raw_code)` and `hooks.fire("phone_setup_started", user=user, phone=..., code=raw_code)` both pass the pre-hash raw code. Neither event is in `KNOWN_EVENTS`, so hosts have no documented warning about the sensitive payload.

**Recommended fix:** Remove `code` from both hook payloads. If hosts need to observe MFA events for monitoring, provide a non-sensitive identifier (e.g., `jti` of the pending token, or a generic "otp issued" boolean flag). Update `KNOWN_EVENTS` to include both events with a note that they do not carry code material.

### W-10 — `record_failure` not called for unverified-account login attempts (enumeration weakness)

**File:** `src/regstack/routers/login.py:86–90`  
**Impact:** An attacker who holds valid credentials for an unverified account receives HTTP 403 (not 401) and the lockout counter is never incremented. This distinguishes "correct password, unverified" from "wrong password", enabling a password-correct oracle against unverified accounts without triggering lockout.

**Recommended fix:** Call `rs.lockout.record_failure(payload.email)` before raising the 403, mirroring how the primary wrong-password path works, or return a generic 401 for all pre-authentication failures.

### W-11 — Missing `permissions:` blocks on build and test workflow jobs

**Files:** `.github/workflows/publish.yml` (`build` job); `.github/workflows/test.yml` (all jobs)  
**Impact:** Without explicit `permissions:`, GitHub falls back to the repository default (often `contents: write`). Any third-party action in these jobs inherits write access they do not need.

The `publish` job correctly declares `permissions: { id-token: write }`. The `build` job and all `test.yml` jobs have no declarations.

**Recommended fix:** Add `permissions: read-all` at the top level of `test.yml` and to the `build` job in `publish.yml`. Add only the minimum necessary grants per-job where write access is genuinely required.

### W-12 — All GitHub Actions `uses:` directives are tag-pinned, not SHA-pinned

**Files:** `.github/workflows/publish.yml`, `.github/workflows/test.yml`  
**Impact:** Tag references can be silently rewritten by the action author. A compromised `actions/checkout` or `astral-sh/setup-uv` tag would execute in the CI/CD environment with access to secrets and the repository.

Actions affected: `actions/checkout@v4` (×4 instances), `astral-sh/setup-uv@v3` (×4), `actions/upload-artifact@v4` (×2), `actions/download-artifact@v4` (×1).

**Recommended fix:** Pin every action to a full commit SHA. Tools like `pin-github-actions` or Dependabot's `actions-version` mode can automate this.

### W-13 — `pywebview`, `uvicorn[standard]`, and `tomlkit` shipped as unconditional base dependencies

**File:** `src/regstack/pyproject.toml` `[project] dependencies`  
**Impact:** Every `pip install regstack` (including production headless server deployments) installs a full GUI WebView framework, an ASGI server, and a TOML library. These packages are used only by the `regstack oauth setup` and `regstack theme design` CLI subcommands. The expanded attack surface includes pywebview's platform-native GUI backends (pyobjc on macOS, PyQt/GTK on Linux), which are large, historically CVE-prone packages irrelevant to authentication.

**Recommended fix:** Move `pywebview>=5.0`, `uvicorn[standard]>=0.29`, and `tomlkit>=0.13` to an optional `wizard` (or `cli`) extra. The `_LazyOauthGroup` / `_LazyThemeGroup` import-time guards already prevent import failures when the packages are absent; removing them from base dependencies would complete the isolation.

### W-14 — `aiosmtplib` minimum constraint is 2 major versions behind the locked version

**File:** `pyproject.toml` `dependencies`: `aiosmtplib>=3.0`; locked: `5.1.0`  
**Impact:** A consumer's dependency resolver could satisfy `>=3.0` with an unmaintained `3.x` release. The `3.x` branch has not received security patches that ship in `5.x`.

**Recommended fix:** Raise the lower bound to `aiosmtplib>=5.0`.

### W-15 — Registration endpoint returns HTTP 409 enabling unauthenticated email enumeration

**File:** `src/regstack/routers/register.py:37–38`  
**Impact:** Any unauthenticated caller can probe whether an email address is registered by attempting to register it. The 409 response body `"An account with that email already exists."` is explicit. This is noted as a deliberate UX decision but contradicts the anti-enumeration posture applied to `forgot-password` (202 regardless) and `resend-verification` (202 regardless).

**Recommended fix:** Return 200 (or 202) with the same body as a successful registration request, relying on the verification email as the signal. Alternatively, document this as an accepted trade-off in `docs/security.md` so the decision is explicit and intentional.

## 🟡 INFO (Best-practice recommendations)

### I-1 — `_encode_email_change_token` emits `iat` as `int`, not `float`

**File:** `src/regstack/routers/account.py:258`  
`int(now.timestamp())` truncates sub-second precision. The CLAUDE.md "Bulk revocation: float iat and <= cutoff" section documents why `iat` must be a float. Email-change tokens are not session tokens and the bulk-revoke check does not apply to them directly, but the convention inconsistency is a maintenance hazard.

---

### I-2 — `TokenError` message from pyjwt echoed verbatim in HTTP 401 `detail`

**File:** `src/regstack/auth/dependencies.py:120`  
`detail=f"Invalid token: {exc}"` can expose which internal JWT validation step failed (e.g., "Signature verification failed", "Token is expired"). Low-severity information leak. Prefer a generic `"Authentication failed."` message.

---

### I-3 — `password_reset_requested`, `verification_requested`, and `email_change_requested` hook payloads include the raw token URL

**Files:** `src/regstack/routers/password.py:67`, `register.py:97`, `account.py:167`  
Host logging hooks that record these events capture live, usable tokens. The risk is lower than I-OTP (raw codes) because the URLs are also in the user's email, but logging hooks are a common audit tool.

**Recommended fix:** Document in `KNOWN_EVENTS` that these payloads contain sensitive token URLs and must not be logged.

Missing from `KNOWN_EVENTS` (but fired in routers): `verification_requested`, `email_change_requested`, `email_changed`, `phone_setup_started`, `mfa_login_started`, `mfa_enabled`, `mfa_disabled`.  
Listed in `KNOWN_EVENTS` but never fired: `user_logged_out` (logout.py fires no hook).

**File:** `src/regstack/routers/oauth.py:153`  
The `was_new` boolean from `_resolve_user` is never stored on the state row and is never returned via the exchange endpoint. SPAs and downstream hook handlers cannot distinguish first-time OAuth registrations from sign-ins, preventing first-login onboarding flows.

### I-6 — Migration `0002_oauth.py` `downgrade()` will fail destructively on deployments with OAuth users

**File:** `src/regstack/backends/sql/migrations/versions/0002_oauth.py`  
The downgrade restores `hashed_password` to `NOT NULL`, which raises a database integrity error if any user has `hashed_password = NULL` (i.e., any OAuth-only account exists). The migration file notes this in a comment but provides no guard. Attempting to roll back `0002` on a production database with OAuth users will leave the migration partially applied.

The `timeout=30` default is buried in `PyJWKClient`'s implementation. The `exchange_code` httpx call uses an explicit `timeout=10.0`. ### I-12 — Unguarded `ObjectId()` calls in 9 write methods in Mongo `UserRepo`

**File:** `src/regstack/backends/mongo/repositories/user_repo.py` (lines 59, 65, 77, 89, 95, 101, 107, 113, 124)  
`get_by_id` and `delete` guard with `ObjectId.is_valid()`. The 9 write methods (`set_last_login`, `set_tokens_invalidated_after`, `update_password`, etc.) call `ObjectId(user_id)` directly. All callers today pass a DB-sourced `id`, so the risk is low in practice, but a future code path that wires an externally-supplied ID to these methods would raise an uncaught `bson.errors.InvalidId` rather than returning a clean error.

### I-13 — `redirect_to` does not normalize backslash or double-slash paths

**File:** `src/regstack/routers/oauth.py:348–360`  
`_validate_redirect()` correctly rejects absolute URLs to other origins. It does not strip leading backslashes (`\evil.com`) or double-slash sequences (`////evil.com`). Browser normalization of these forms is client-dependent. While the same-origin check catches most cases, normalizing the path before validation would close any browser-specific gaps.

### I-14 — `oauth` extra allows transitive `cryptography` versions with open CVEs

**File:** `pyproject.toml` `[project.optional-dependencies]` `oauth = ["pyjwt[crypto]>=2.8"]`  
The locked `cryptography 47.0.0` is patched for CVE-2026-26007 (CVSS 8.2, small-subgroup key leak), CVE-2026-34073 (DNS name constraint bypass), and CVE-2026-39892 (buffer overflow on Python ≥3.11). A consumer whose resolver satisfies `pyjwt[crypto]>=2.8` with an older `cryptography` build would be exposed. Adding an explicit floor `cryptography>=46.0.7` in the `oauth` extra protects downstream consumers.

### I-15 — CI `publish.yml` `workflow_dispatch` trigger allows arbitrary-branch artifact builds

**File:** `.github/workflows/publish.yml`  
`workflow_dispatch` is listed as a trigger. The `build` job runs unconditionally on manual dispatch; the `publish` job is correctly guarded by `if: startsWith(github.ref, 'refs/tags/v')`. A repo-write-access holder can trigger a build from any branch and produce a dist artifact, though they cannot publish it to PyPI. This is low risk but noted for completeness.
