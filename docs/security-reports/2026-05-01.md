## 🟠 WARNING

### W-1 · JWT decode error details leaked to API clients
**File:** `src/regstack/auth/dependencies.py:120`  
**Section:** §9 (Data Exposure), §7 (Session & Token Security)

```python
raise HTTPException(
    status_code=status.HTTP_401_UNAUTHORIZED,
    detail=f"Invalid token: {exc}",   # ← leaks pyjwt error message
    headers={"WWW-Authenticate": "Bearer"},
) from exc
```

`TokenError` wraps the raw `pyjwt.PyJWTError` message and re-surfaces it verbatim in the HTTP response body. A caller can observe strings like `"Invalid token: Signature verification failed"`, `"Invalid token: Signature has expired"`, or `"Invalid token: token purpose mismatch"`. The spec (§9) and `CLAUDE.md` are explicit: JWT decode errors must return a generic 401 with no specific reason.

**Fix:** replace the f-string with a static message:
```python
detail="Authentication credentials are invalid.",
```

### W-2 · Admin `DELETE /admin/users/{id}` does not cascade OAuth identities
**File:** `src/regstack/routers/admin.py:131-134`  
**Section:** §3 (Auth & Authz Audit)

The user-initiated `DELETE /account` path (in `account.py`) correctly calls `await rs.oauth_identities.delete_by_user_id(user.id)` before deleting the user row. The admin-initiated `DELETE /admin/users/{user_id}` path does **not**:

```python
async def delete_user(...):
    user = await _require_user(rs, user_id)
    assert user.id is not None
    await rs.users.delete(user.id)            # ← deletes user first
    await rs.pending.delete_by_email(user.email)
    await rs.lockout.clear(user.email)
    # missing: await rs.oauth_identities.delete_by_user_id(user.id)
```

This leaves orphaned `oauth_identities` rows pointing at a now-deleted `user_id`. While MongoDB ObjectId / UUID4 recycling is highly unlikely, the orphaned rows represent a parity violation, consume storage, and could confuse audit tooling. The spec (§3) requires "Account deletion must cascade `oauth_identities` first (anti-orphan), then delete the user."

**Fix:** Add the cascade before `rs.users.delete`:
```python
await rs.oauth_identities.delete_by_user_id(user.id)
await rs.users.delete(user.id)
```

### W-3 · GitHub Actions: third-party actions pinned to mutable version tags, not commit SHAs
**Files:** `.github/workflows/publish.yml`, `.github/workflows/test.yml`  
**Section:** §12a (Third-party action pinning)

All `uses:` directives reference floating version tags that can be rewritten by the action author without changing the tag:

| Workflow | Action | Current ref |
|----------|--------|-------------|
| `publish.yml`, `test.yml` | `actions/checkout` | `@v4` |
| `publish.yml`, `test.yml` | `astral-sh/setup-uv` | `@v3` |
| `publish.yml`, `test.yml` | `actions/upload-artifact` / `download-artifact` | `@v4` |
| `publish.yml` | `pypa/gh-action-pypi-publish` | `@release/v1` |

`pypa/gh-action-pypi-publish` runs with `id-token: write` permission and directly publishes to PyPI — it is the highest-risk action in this repository. A tag-rewrite supply-chain attack on this action could publish a malicious wheel.

**Fix:** Pin every action to its commit SHA:
```yaml
uses: actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683  # v4
uses: astral-sh/setup-uv@<sha>  # v3
uses: pypa/gh-action-pypi-publish@<sha>  # release/v1
```
Use `@v4` (or similar) as a comment for readability. Tools like `pin-github-action` or Renovate can automate this.

### W-4 · JWT secret minimum length not enforced
**File:** `src/regstack/auth/jwt.py:118`  
**Section:** §7 (Session & Token Security)

`JwtCodec.__init__` only rejects an **empty** `jwt_secret`:

```python
if not config.jwt_secret.get_secret_value():
    raise ValueError("RegStackConfig.jwt_secret is empty ...")
```

The spec and `CLAUDE.md` both state "≥32 chars enforced at construction." A secret shorter than 32 bytes provides inadequate HMAC-SHA256 strength and leaves derived keys weak. There is no validator in `RegStackConfig._warn_empty_secret` that enforces a length floor.

**Fix:** Enforce a 32-character minimum at `JwtCodec` construction:
```python
secret = config.jwt_secret.get_secret_value()
if len(secret) < 32:
    raise ValueError(
        "RegStackConfig.jwt_secret must be at least 32 characters. "
        "Run `regstack init` to generate one."
    )
```

### W-5 · `PyJWKClient` for Google JWKS has no explicit network timeout
**File:** `src/regstack/oauth/providers/google.py:82`  
**Section:** §11 (Third-Party Integration Security)

```python
self._jwks_client = PyJWKClient(jwks_url, cache_keys=True)
```

`PyJWKClient` fetches the JWKS URL synchronously via `urllib.request.urlopen` on cache misses. Without a `timeout=` argument, Python's default applies, which is no timeout on some platforms. A slow or unreachable JWKS endpoint will block the OAuth callback indefinitely, creating a DoS vector for any attacker who can degrade connectivity to `www.googleapis.com/oauth2/v3/certs`.

The `httpx.AsyncClient` for token exchange is correctly limited to `timeout=10.0`. The JWKS fetch should be treated the same way.

**Fix:**
```python
self._jwks_client = PyJWKClient(jwks_url, cache_keys=True, timeout=10)
```

### I-2 · GitHub Actions workflows missing explicit `permissions:` on non-publishing jobs
**Files:** `.github/workflows/publish.yml` (`build` job), `.github/workflows/test.yml` (all jobs)  
**Section:** §12c (Workflow permissions)

The `publish` job in `publish.yml` correctly declares `permissions: { id-token: write }`. The sibling `build` job and all jobs in `test.yml` have no `permissions:` block — they inherit the repository default. Unless the repository's default is set to `read-only` (Settings → Actions → Workflow permissions), these jobs receive broader-than-necessary permissions.

**Fix:** Add a minimal permissions block to each job:
```yaml
permissions:
  contents: read
```

### I-3 · OAuth provider name exposed in 404 response on public `start` endpoint
**File:** `src/regstack/routers/oauth.py:343`  
**Section:** §9 (Data Exposure)

```python
raise HTTPException(
    status_code=status.HTTP_404_NOT_FOUND,
    detail=f"OAuth provider {name!r} is not configured.",
)
```

`GET /oauth/{provider_name}/start` is a public endpoint. An invalid provider name receives a 404 whose body echoes the caller-supplied `provider_name`. This is low-risk (provider names are not secret and the response is 404), but it allows an attacker to confirm which provider names the server recognises vs. which it doesn't.

**Fix (optional):** Return a generic message: `"OAuth provider not found."` or simply drop the provider name from the string.

### I-4 · Rate-limit config fields declared but not implemented
**File:** `src/regstack/config/schema.py:147-149`  
**Section:** §6 (Rate Limiting)

`login_max_per_minute` and `login_max_per_hour` appear in `RegStackConfig` with range validators, but no code enforces them — only `login_lockout_threshold` and `login_lockout_window_seconds` are used. Hosts reading the configuration may assume per-minute throttling is active when it is not.

**Recommendation:** Either remove the fields until they are implemented, or add a docstring clearly marking them `# Not yet implemented — see issue #X`.

### I-5 · MongoDB Server CVE-2025-14847 ("MongoBleed") — check server version
**Section:** §1 (Dependency CVE Check)

PyMongo 4.17.0 (the driver) has no known CVEs. However, MongoDB Server CVE-2025-14847 was disclosed and affects server versions before 8.2.3 / 8.0.17 / 7.0.28 / 6.0.27. This is a server-side vulnerability, not a driver issue. Hosts using the MongoDB backend should confirm their server is on a patched release.

### I-6 · PyPI sigstore wheel attestations not confirmed
**Section:** §13e (Supply-chain bill of materials)

`pypa/gh-action-pypi-publish` supports `attestations: true` to sign the wheel via sigstore. The current `publish.yml` does not explicitly set this flag. Without wheel attestations, consumers cannot cryptographically verify that a given wheel was produced by this GitHub Actions workflow.

**Fix (optional):** Add to the publish step:
```yaml
- uses: pypa/gh-action-pypi-publish@<sha>
  with:
    attestations: true
```

## 🟢 CLEAN

The following areas were reviewed in full and found to be secure:

**§1 Dependency CVEs (Nov 2025 – May 2026):** All locked dependencies are on versions with no known open CVEs. Notably: jinja2 3.1.6 resolves CVE-2025-27516 (sandbox breakout) and python-multipart 0.0.26 resolves CVE-2026-40347 (DoS) — both already on the fixed versions.

**§2 Hardcoded Secrets:** No API keys, private keys, JWTs, or credential strings found in `src/`, `tests/`, `examples/`, or `docs/`. Example TOML files contain no `jwt_secret` values. Git history shows no accidentally committed secret files.

**§3 Auth & Authz:**
- All authenticated endpoints correctly inject `rs.deps.current_user()` or `rs.deps.current_admin()`.
- `UserPublic.from_user()` is used in every user-data response; `hashed_password` and `tokens_invalidated_after` are never exposed.
- Bulk-revoke discipline: `update_password` bumps `tokens_invalidated_after` (password change, reset); `update_email` bumps it (email change confirm); admin disable sets it via `set_tokens_invalidated_after`.
- `is_payload_bulk_revoked` uses `payload.iat <= cutoff` (conservative `<=` comparison) and `JwtCodec.encode` correctly emits `iat` as a float.
- `JwtCodec.decode` disables pyjwt's clock-based `exp`/`iat` checks and re-validates both against the injected `Clock`.
- OAuth-only user (`hashed_password=None`) returns the same generic 401 as a wrong-password attempt; `change-password`, `change-email`, `delete-account` return 400 with a forgot-password pointer.
- Per-purpose JWT key derivation via HMAC-SHA256 (`derive_secret`) correctly isolates purposes.

**§4 Storage-layer Injection:**
- MongoDB repos use parameterized pymongo driver calls; no `$where`, `$expr`, `$function`, or `$accumulator` operators; no user-input regex patterns.
- `ObjectId.is_valid(user_id)` guards every path that parses user IDs as ObjectIds.
- SQL repos use SQLAlchemy Core parameterized binds exclusively; the only `text()` call is a health-check `SELECT 1` with no user input.
- `UtcDateTime` TypeDecorator correctly re-attaches UTC `tzinfo` on reads.

**§5 Input Validation:**
- Every endpoint body uses a Pydantic `BaseModel` with `extra="forbid"`.
- `EmailStr` on all email fields; `is_valid_e164()` on all phone inputs.
- Jinja2 email templates use `autoescape=select_autoescape(["html"])`.
- SSR UI templates: no `| safe` filters, no `{% autoescape false %}` blocks.
- CLI `init` wizard writes files directly; no shell command construction.
- OAuth `redirect_to` is validated same-origin against `config.base_url` (`_validate_redirect`).

**§6 Rate Limiting:**
- `LockoutService.check()` is called at the top of the login handler, before any password verification — a locked account returns 429 immediately.
- `record_failure` is called on every wrong-credential path (unknown user, wrong password, OAuth-only account).
- Admin `/admin/users` endpoint caps `limit` at 200 (`Query(ge=1, le=200)`).
- OAuth state TTL defaults to 300 s; Mongo TTL index and SQL `purge_expired()` both present.

**§8 CSP and SSR Layer:**
- All UI templates (`base.html`, `login.html`, `register.html`, `forgot.html`, `reset.html`, `verify.html`, `me.html`, `mfa_confirm.html`, `email_change_confirm.html`, `oauth_complete.html`) contain no `<style>` blocks, no `style="..."` attributes, and no inline `<script>` tags.
- The only `<script>` is `<script src="{{ static_prefix }}/js/regstack.js" defer>` — external, no inline code.
- Pages are stateless: no user data (email, name, id) rendered into HTML.

**§9 Data Exposure (partial):**
- Database driver errors are not surfaced to clients.
- OAuth callback failures redirect to `/login?error=<short-code>`; specific exception text is only logged server-side.
- No logging of JWT secret, OAuth client_secret, or SMTP password found.

**§10 Git History:** No sensitive files committed; no secret strings in commit diffs.

**§11 Third-Party Integrations:**
- SMTP: password via `SecretStr.get_secret_value()`, never logged.
- SES: region/profile via config, never hardcoded.
- SNS/Twilio: credentials via config, lazy imports.
- Google OAuth: `client_secret` stored as plain `str` in `GoogleProvider` (passed by the host at construction from `OAuthConfig.google_client_secret: SecretStr`); ID-token verification validates signature (JWKS), `iss`, `aud`, `exp`, `nonce`.

**§12 GitHub Actions (partial):** No `pull_request_target` triggers; no self-hosted runners; no `PYPI_API_TOKEN` secrets; no user-controlled input in `run:` blocks; cache keys are not user-controlled.

**§13 PyPI Package Integrity:**
- `[tool.hatch.build.targets.wheel]` scopes the wheel to `src/regstack` — no test fixtures, docs, or build artifacts included.
- `pyproject.toml` `dependencies` list contains only base-install packages; `pymongo`, `asyncpg`, `aioboto3`, `twilio`, `pyjwt[crypto]` are correctly in `[project.optional-dependencies]`.
- Version read from `pyproject.toml` via `tomllib` in the publish workflow (no package import, no 0.2.0-style regression).
- `version.py` and `pyproject.toml` are in sync at `0.3.0`.

**§14 OAuth Subsystem:**
- PKCE: `code_verifier` stored in the state row server-side; only `code_challenge` (SHA-256 hash) goes to the provider.
- State row lifecycle: `find_one_and_delete` (Mongo) / atomic read+delete in one transaction (SQL) for `consume`; expiry check uses `rs.clock.now()`.
- `verify_id_token` validates: JWKS signature, `iss`, `aud`, `exp`, `nonce`, non-empty `sub`.
- Linking policies: auto-link only when `auto_link_verified_emails=True AND email_verified=true`; `identity_in_use`, `already_linked`, `email_in_use` error codes correct.
- `(provider, subject_id)` and `(user_id, provider)` unique constraints enforced at DB level on both Mongo (unique indexes) and SQL (`UniqueConstraint`).
- All four OAuth hook events (`oauth_signin_started`, `oauth_signin_completed`, `oauth_account_linked`, `oauth_account_unlinked`) in `KNOWN_EVENTS` and fired at correct points in the router.

**§15 Multi-backend Parity:**
- `test.yml` CI matrix runs all three backends (SQLite, Postgres, MongoDB) on every push.
- Migration `0002_oauth` has working `upgrade()` and `downgrade()`; uses `batch_alter_table` for SQLite compatibility.
- Mongo index set matches SQL unique constraints for `oauth_identities`.
- `tz_aware=True` for Mongo client; `UtcDateTime` TypeDecorator for SQL.
