## 🟠 WARNING (address within 1 week)

### W-1 · JWT decode reason leaked into HTTP 401 response

**File:** `src/regstack/auth/dependencies.py:120`

```python
detail=f"Invalid token: {exc}",
```

`TokenError` is propagated verbatim into the HTTP response body. Messages such as
`"Signature has expired"`, `"token purpose mismatch"`, and raw pyjwt error strings
are returned to the client. The review spec is explicit: *"JWT decode errors → 401
generic, never the specific reason."* An expired-token message confirms to an
attacker that the token structure and signature are valid (only the clock differs),
which aids replay attempts and debugging of token-theft attacks.

**Fix:** Replace with a static detail string, e.g.
`detail="Authentication required."`. The raw reason can be emitted at `DEBUG` level
in the server log.

### W-2 · Admin `DELETE /admin/users/{id}` does not cascade `oauth_identities`

**File:** `src/regstack/routers/admin.py:126-135`

The user self-service `DELETE /account` correctly calls
`await rs.oauth_identities.delete_by_user_id(user.id)` before removing the user.
The admin deletion endpoint omits this step:

```python
async def delete_user(...):
    user = await _require_user(rs, user_id)
    await rs.users.delete(user.id)       # ← oauth_identities not cleaned up
    await rs.pending.delete_by_email(user.email)
    ...
```

After admin deletion, orphan rows with the deleted `user_id` remain in
`oauth_identities`. Neither the Mongo backend (no FK) nor the SQL backend (no
`ForeignKeyConstraint`) enforces referential integrity here, so the rows silently
accumulate. The review spec section §3 is explicit: *"Account deletion must cascade
`oauth_identities` first (anti-orphan), then delete the user."*

**Fix:** Add `await rs.oauth_identities.delete_by_user_id(user.id)` before
`await rs.users.delete(user.id)` in `delete_user`.

### W-3 · GitHub Actions: Third-party actions pinned to mutable version tags, not SHAs

**Files:** `.github/workflows/publish.yml`, `.github/workflows/test.yml`

Every `uses:` directive references a moving tag:

| Workflow | Action | Tag |
|---|---|---|
| publish.yml | `actions/checkout` | `@v4` |
| publish.yml | `astral-sh/setup-uv` | `@v3` |
| publish.yml | `actions/upload-artifact` | `@v4` |
| publish.yml | `actions/download-artifact` | `@v4` |
| publish.yml | `pypa/gh-action-pypi-publish` | `@release/v1` |
| test.yml | `actions/checkout` | `@v4` |
| test.yml | `astral-sh/setup-uv` | `@v3` |
| test.yml | `actions/upload-artifact` | `@v4` |

Version tags can be force-pushed by the action maintainer. `pypa/gh-action-pypi-publish`
is the highest-value target: it runs with `id-token: write` in the `pypi` environment
and has the capability to publish arbitrary wheels to PyPI. A compromised or hijacked
tag on that action would silently publish a backdoored `regstack` wheel to every
downstream `pip install` consumer.

**Fix:** Pin every action to a commit SHA, e.g.
`pypa/gh-action-pypi-publish@76f52bc884231f62e01b7ae2b9b48f2f5a4c68b`.
Re-pin after intentional upgrades. Tools such as `pin-github-actions` automate this.

### W-4 · GitHub Actions: Missing `permissions:` declarations

**Files:** `.github/workflows/test.yml` (entire workflow), `.github/workflows/publish.yml`
(the `build` job)

Neither the `test.yml` workflow nor the `build` job in `publish.yml` declare explicit
`permissions:`. When not declared at the job level, the effective permissions fall
through to the repository default (organisation-dependent; often
`contents: read, pull-requests: write` or wider). The `publish` job correctly
declares `permissions: { id-token: write }`.

**Fix:** Add a minimal `permissions:` block to every job:

```yaml
# test.yml jobs and publish.yml build job
permissions:
  contents: read
```

### W-5 · Google JWKS client constructed without an explicit network timeout

**File:** `src/regstack/oauth/providers/google.py:82`

```python
self._jwks_client = PyJWKClient(jwks_url, cache_keys=True)
```

`PyJWKClient` internally calls `urllib.request.urlopen` with no default timeout.
If Google's JWKS endpoint (`https://www.googleapis.com/oauth2/v3/certs`) is
unreachable or slow, the call blocks indefinitely, hanging the request worker for
the duration of the OAuth callback. Under targeted network interference this becomes
a partial denial of service against the OAuth sign-in path.

The `PyJWKClient` constructor accepts a `timeout` keyword argument since PyJWT 2.6.

**Fix:**

```python
self._jwks_client = PyJWKClient(jwks_url, cache_keys=True, timeout=10)
```

### I-3 · `aiosmtplib` is a base-install dependency despite being optional-backend code

**File:** `pyproject.toml` (base `dependencies` list)

`aiosmtplib>=3.0` ships with every `pip install regstack`, including users who
configure `email.backend = "console"` or `"ses"`. The library is imported lazily
in `src/regstack/email/smtp.py` only when the SMTP backend is selected, so there is
no import-time regression — but it adds supply-chain surface area for consumers who
will never use SMTP.

**Recommendation:** Move `aiosmtplib` to an optional extra (e.g. `smtp`) alongside
`ses = ["aioboto3>=..."]`. Update docs to show `pip install regstack[smtp]`.
This is a semver-significant change; defer to a minor release.

### I-4 · Wheel publication does not use sigstore attestations

**File:** `.github/workflows/publish.yml`

`pypa/gh-action-pypi-publish` supports build provenance attestations via
`attestations: true`. Without it, the published wheel has no cryptographically
verifiable link back to this repository or this CI run. Sigstore / PEP 740
attestations are now surfaced directly on PyPI package pages.

**Recommendation:** Add `attestations: true` to the publish step when upgrading to
a SHA-pinned version of the action (see W-3).

### I-5 · `OAuthConfig.completion_ttl_seconds` is declared but never applied

**File:** `src/regstack/config/schema.py:71`, `src/regstack/routers/oauth.py`

`OAuthConfig` defines `completion_ttl_seconds: int = 30`, described as the window
after the callback during which the SPA must call `/oauth/exchange`. The value is
never read by the router; the state row retains its full `state_ttl_seconds` (5
minutes) even after the callback writes `result_token`. The effective post-callback
window is therefore 5 minutes, not 30 seconds.

While the security impact is low (the `state_id` is 256-bit random, and `consume`
is atomic), the config field misleads operators who set it expecting tighter post-
callback enforcement.

**Recommendation:** Either implement the shorter TTL on `set_result_token` (update
`expires_at` to `now + completion_ttl_seconds`), or remove the config field and
document that the full `state_ttl_seconds` applies for the exchange window.

### I-6 · `ExchangeResponse.was_new_account` is always `False`

**File:** `src/regstack/routers/oauth.py:151-155`

`_resolve_user()` correctly returns `(user, was_new)`, but `was_new` is never
persisted to the state row. The `/oauth/exchange` response therefore always reports
`was_new_account=False` regardless of whether a new account was created. Client code
that tries to show a "Welcome!" first-login experience based on this field is silently
broken.

**Fix:** Add a `was_new_account: bool` column to the `oauth_states` table/document
and persist the value in `set_result_token`.

### §2 — Hardcoded Secrets

No API keys, JWTs, connection-string credentials, or private keys found in `src/`,

### §3 — Authentication & Authorisation

- Every authenticated endpoint injects `Depends(rs.deps.current_user())` or
  `Depends(rs.deps.current_admin())`.
- `UserPublic` projection is used uniformly; `hashed_password` and
  `tokens_invalidated_after` are never serialised to API responses.
- Bulk-revoke discipline verified: password reset calls `update_password` (bumps
  `tokens_invalidated_after`) **and** `lockout.clear`; password change does the
  same; email-change confirm calls `update_email` which bumps
  `tokens_invalidated_after`; admin disable calls `set_tokens_invalidated_after`.
- `BaseUser.hashed_password is None` case handled correctly: same generic 401 as
  wrong password; `change-password`, `change-email`, `delete-account` all call
  `_require_password_set` which returns 400 with a reset-flow pointer.
- Constant-time password comparison via `pwdlib` (no `==`).
- JWT keys derived per-purpose via HMAC-SHA256; compromise of one derived key does
  not compromise the master.
- **Admin `delete_user` cascade gap** — see W-2.

### §4 — Storage-Layer Injection

**MongoDB:** No user input is interpolated into query filter dicts. No
`$where`, `$expr`, `$function`, or `$accumulator` operators used. No regex built
from user input. All `ObjectId` path params are guarded by `ObjectId.is_valid()`
before use. `$in` queries do not accept user-supplied arrays.

**SQL:** All queries use SQLAlchemy Core with bound parameters. The single
`text("SELECT 1")` call in the backend ping is a fixed literal with no user
input. `UtcDateTime.process_result_value` re-attaches UTC on read, preventing
tz-naive regression in bulk-revoke comparisons. `purge_expired()` implementations
use bound `cutoff` parameters throughout.

### §5 — Input Validation

- All API endpoints accept typed Pydantic models with `extra="forbid"`.
- `EmailStr` validation on all email fields.
- `is_valid_e164()` called before storage in the phone-setup route.
- Jinja2 email templates: `autoescape` on for HTML; the `| safe` occurrence in the
  email templates is in a plain-text literal sentence ("you can safely ignore"),
  not a Jinja filter.
- SSR UI templates (`src/regstack/ui/templates/`): no `| safe` filters, no
  `{% autoescape false %}`, no `style="..."` attributes, no inline `<script>` tags.
- The `regstack init` CLI wizard uses `click.prompt()` exclusively; no shell
  interpolation.
- `redirect_to` in OAuth `start` and `link/start` is validated same-origin against
  `config.base_url` via `_validate_redirect()`.

### §6 — Rate Limiting & DoS

- Login lockout `check()` runs before password verification; locked-out callers
  receive 429 without the password being tested.
- `record_failure` is called for non-existent accounts and wrong-password attempts.
- MongoDB: TTL index on `login_attempts.when` sized to `login_lockout_window_seconds`.
- SQL: `count_recent` filters on `when >= now - window`; `purge_expired` reaps old rows.
- Admin user listing caps `limit` at 200 via `Query(ge=1, le=200)`.
- OAuth state rows: Mongo TTL index on `expires_at`; SQL `purge_expired` reaper.
- No file upload endpoints added.

### §7 — Session & Token Security

- `_authenticate` chain: decode → blacklist → bulk revoke → `is_active`. All four

- No token values appear in `log.info`/`log.error` calls (OAuth errors log short

### §14 — OAuth Subsystem

- PKCE: `code_verifier` generated server-side and stored on the `oauth_states` row;
  only the SHA-256 `code_challenge` goes to the provider. Verified in `_begin_flow`.
- State lifecycle: missing row → `bad_state`; expired → `state_expired` (compared
  via `rs.clock.now()`); exchange is atomic via `find_one_and_delete` (Mongo) /
  `SELECT + DELETE` in one transaction (SQL).
- ID-token verification: signature via JWKS, `iss`, `aud`, `exp`, `nonce` all
  validated; failures raise `OAuthIdTokenError` with generic outer message logged
  at WARNING; raw exception not forwarded to redirect.
- Linking policy: `auto_link_verified_emails` off by default; already-linked
  identity to different user → `identity_in_use`; same identity re-link →
  `already_linked`.
- OAuth-only user paths: same 401 on password login; 400 on
  change-password/change-email/delete-account; unlink refused when last auth method.
- Identity uniqueness: `(provider, subject_id)` and `(user_id, provider)` have
  unique constraints at both Mongo (index) and SQL (`UniqueConstraint`) level.
- Hook events: `oauth_signin_started`, `oauth_signin_completed`,
  `oauth_account_linked`, `oauth_account_unlinked` all in `KNOWN_EVENTS` and fired
  at the correct points.
- **`completion_ttl_seconds` dead config** — see I-5.
- **`was_new_account` always False** — see I-6.

### §15 — Multi-Backend Parity & Schema Migration Safety

- CI matrix runs pytest against SQLite, MongoDB, and Postgres on every push (both
  Python 3.11 and 3.12). Satisfies the CLAUDE.md "A test run that doesn't cover
  every backend is a failing test" rule.
- Migration `0002_oauth.py` uses `batch_alter_table` for the `hashed_password`
  nullable change; both `upgrade()` and `downgrade()` are implemented.
- Mongo unique indexes match every SQL `UniqueConstraint` for `users.email`,
  `(provider, subject_id)`, and `(user_id, provider)`.
- TTL handling: Mongo TTL indexes cover `blacklist.exp`, `pending.expires_at`,
  `mfa_codes.expires_at`, `oauth_states.expires_at`, `login_attempts.when`; SQL
  repos provide `purge_expired()` with host-documented reaper contract.
- `MongoClient` constructed with `tz_aware=True`; `UtcDateTime` TypeDecorator
  re-attaches UTC on read.

### Top 3 priorities for the team

1. **(W-1) Strip the JWT error reason from 401 responses.** One-line fix in
   `auth/dependencies.py:120`. No tests should break — the error type (401) is
   what matters, not the detail string.

   add in `routers/admin.py`. Prevents orphan identity rows that accumulate silently
   on every admin-initiated user deletion.

3. **(W-3) Pin GitHub Actions to commit SHAs.** The PyPI publish action runs with
   `id-token: write` and is the highest-impact supply-chain target in this
   repository. A compromised `@release/v1` tag can publish a backdoored wheel to
   every downstream consumer.
