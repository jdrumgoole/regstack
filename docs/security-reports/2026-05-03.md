## 🟠 WARNING (Address within 1 week)

### W-1 — GitHub Actions: all third-party actions pinned to mutable tags, not commit SHAs

**Files:** `.github/workflows/publish.yml`, `.github/workflows/test.yml`
**Section:** §12a

Every `uses:` directive in both workflows references a mutable tag or branch ref:

| Workflow | Action | Ref used | Risk |
|---|---|---|---|
| publish.yml | `actions/checkout` | `@v4` | Tag can be repointed |
| publish.yml | `astral-sh/setup-uv` | `@v3` | Tag can be repointed |
| publish.yml | `actions/upload-artifact` | `@v4` | Tag can be repointed |
| publish.yml | `actions/download-artifact` | `@v4` | Tag can be repointed |
| publish.yml | `pypa/gh-action-pypi-publish` | `@release/v1` | **Branch ref** — drifts silently |
| test.yml | `actions/checkout` | `@v4` | Tag can be repointed |
| test.yml | `astral-sh/setup-uv` | `@v3` | Tag can be repointed |
| test.yml | `actions/upload-artifact` | `@v4` | Tag can be repointed |

`pypa/gh-action-pypi-publish@release/v1` is the highest-risk entry: it runs with
`id-token: write` access (OIDC trusted publisher) and pushes the wheel to PyPI. A
branch reference means any commit pushed to `release/v1` by the action maintainer
is silently executed with production secrets access.

**Fix:** Pin every `uses:` to a full commit SHA. Use a tool like `pin-github-actions`
or `renovatebot` to keep SHAs current:

```yaml
- uses: actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683  # v4.2.2
- uses: pypa/gh-action-pypi-publish@76f52bc884231f62e3be0e66b12b5b72fa5f6f4b  # release/v1
```

### W-2 — GitHub Actions: missing `permissions:` on build job and all test jobs

**Files:** `.github/workflows/publish.yml` (build job), `.github/workflows/test.yml` (all jobs)
**Section:** §12c

The `publish.yml` publish job correctly declares `permissions: { id-token: write }`.
The build job in the same workflow, and every job in `test.yml`, have no
`permissions:` declaration at all. Without a declaration the job inherits the
repo-level default which, for public repos, defaults to `read` for most scopes
but `write` for `contents` in some GitHub plan configurations.

**Fix:** Add a minimal `permissions:` block to every job:

```yaml
permissions:
  contents: read
```

The build job in `publish.yml` needs `contents: read` only. The test jobs need
`contents: read`; the services (Mongo, Postgres) need no extra permissions.

### W-3 — JWT decode error detail leaks the specific failure reason to callers

**File:** `src/regstack/auth/dependencies.py:118`
**Section:** §7, §9

`_authenticate` raises:
```python
raise HTTPException(
    status_code=status.HTTP_401_UNAUTHORIZED,
    detail=f"Invalid token: {exc}",   # ← leaks the TokenError message
    ...
)
```

`TokenError` wraps `pyjwt.PyJWTError`, whose `.args[0]` carries strings like
`"Signature has expired"`, `"Signature verification failed"`, `"Not enough
segments"`, and `"token purpose mismatch"`. An unauthenticated caller can
distinguish an expired-but-valid token from a tampered/crafted one. This
information can accelerate token-replay and forgery attempts.

The review specification says: *"JWT decode errors → 401 generic, never the
specific reason."*

**Fix:** Return a constant detail string:
```python
raise HTTPException(
    status_code=status.HTTP_401_UNAUTHORIZED,
    detail="Authentication required.",
    headers={"WWW-Authenticate": "Bearer"},
) from exc
```
Log the original `exc` at DEBUG level if diagnostics are needed server-side.

### W-4 — Admin-initiated user deletion does not cascade `oauth_identities`

**File:** `src/regstack/routers/admin.py:125–135`
**Section:** §3

The admin `DELETE /admin/users/{id}` endpoint:
```python
await rs.users.delete(user.id)
await rs.pending.delete_by_email(user.email)
await rs.lockout.clear(user.email)
```

The user-facing `DELETE /account` endpoint (account.py:233) correctly cascades:
```python
await rs.oauth_identities.delete_by_user_id(user.id)   # ← missing in admin
await rs.users.delete(user.id)
```

When an admin deletes a user the `oauth_identities` rows survive, pointing at a
non-existent `user_id`. Because user IDs (UUIDs / ObjectIds) are never reused the
immediate takeover risk is negligible, but the orphan rows represent corrupt state
and violate the explicit cascade requirement stated in CLAUDE.md.

**Fix:** Add `await rs.oauth_identities.delete_by_user_id(user.id)` immediately
before `await rs.users.delete(user.id)` in the admin delete handler, mirroring the
user-facing endpoint.

### W-5 — `pyproject.toml` lower bound for `pyjwt` is below the CVE-2026-32597 fix

**File:** `pyproject.toml:16`
**Section:** §1, §13c

`pyproject.toml` declares `pyjwt>=2.8`. CVE-2026-32597 (CVSS 7.5, published
2026-03-13) affects all pyjwt versions before 2.12.0. The vulnerability is
improper validation of the `crit` (Critical) Header Parameter in JWS tokens —
tokens containing unrecognised critical extensions are accepted instead of
rejected, violating RFC 7515 §4.1.11.

The `uv.lock` file currently pins pyjwt **2.12.1** (patched), so the installed
environment is safe. However, a host who resolves regstack's dependencies against
an older resolver state, or who explicitly constrains `pyjwt<2.12`, would install
a vulnerable version. The published package metadata on PyPI (`>=2.8`) provides no
protection.

**Fix:** Bump the lower bound to `pyjwt>=2.12` in `pyproject.toml`:
```toml
"pyjwt>=2.12",
```
Also update the `oauth` extra: `"pyjwt[crypto]>=2.12"`.

### W-6 — Wizard-only dependencies bundled in base package

**File:** `pyproject.toml:25–29`
**Section:** §1, §13c

Three heavy packages appear in the base `dependencies` list:
```toml
"pywebview>=5.0",
"tomlkit>=0.13",
"uvicorn[standard]>=0.29",
```

These are used solely by the `regstack oauth setup` CLI wizard. Every downstream
consumer who runs `pip install regstack` installs a native GUI webview framework
(`pywebview`), a production ASGI server (`uvicorn[standard]`), and a TOML editor
(`tomlkit`) even when they never run the wizard. This unnecessarily expands the
installed attack surface of library consumers and conflicts with the stated
principle that optional extras must not be forced on all consumers (see §13c of the
review prompt and the 0.2.1 regression history).

Note: `uvicorn[standard]` also appears in the `dev` extra, so it would be installed
in development regardless.

**Fix:** Move `pywebview`, `tomlkit`, and the wizard-specific `uvicorn` to an
optional `wizard` extra:
```toml
[project.optional-dependencies]
wizard = ["pywebview>=5.0", "tomlkit>=0.13", "uvicorn[standard]>=0.29"]
```
Guard the lazy import in `cli/__main__.py` so `regstack init` / `doctor` still work
without the `wizard` extra, and document `pip install regstack[wizard]` in the
OAuth setup CLI reference.

## 🟡 INFO (Best practice recommendations)

### I-1 — Raw OTP codes passed through public hook events

**Files:** `src/regstack/routers/login.py:158`, `src/regstack/routers/phone.py:103`
**Section:** §11

The `mfa_login_started` hook is fired with `code=raw_code` (the 6-digit OTP before
hashing), and `phone_setup_started` is fired with both `code=raw_code` and
`phone=payload.phone_number`. Host implementations that log every hook event
wholesale (e.g., via a generic audit logger) will inadvertently log one-time codes
in plaintext, reducing their security from TOTP-grade to log-access-grade.

**Recommendation:** Document the security sensitivity of the `code` argument
prominently in the hooks reference. Consider omitting `code` from
`mfa_login_started` and providing it only through the SMS delivery mechanism
itself; if hosts need the code for custom delivery, add a separate dedicated hook
(`mfa_code_generated`) that is clearly labelled as carrying a secret value.

### I-2 — `OAuthConfig.completion_ttl_seconds` declared but never used

**File:** `src/regstack/config/schema.py:71`
**Section:** §14

`completion_ttl_seconds: int = 30` is declared in `OAuthConfig` but is not
referenced anywhere in the router, repos, or tests. The exchange window between
callback completion and token exchange is currently governed by the state row's
full `expires_at` (5 minutes). The intent appears to be a shorter post-callback
window to narrow the time an attacker who steals the `state_id` (after the callback
URL appears in access logs) has to exchange it.

**Recommendation:** Either wire `completion_ttl_seconds` into
`oauth_states.set_result_token` (update `expires_at = now + completion_ttl_seconds`
at callback completion time), or remove the field to avoid implying a guarantee that
isn't enforced.

### I-3 — `ExchangeResponse.was_new_account` hardcoded to `False`

**File:** `src/regstack/routers/oauth.py:150–155`
**Section:** §14

The `/oauth/exchange` response always returns `was_new_account=False` because the
state row doesn't persist whether `_resolve_user` created a new account. The SSR
`oauth-complete.html` page uses JS to distinguish first-time vs. returning users for
UX messaging. Since the SPA drives the redirect, no security control depends on this
field, but it's dead/misleading API surface.

**Recommendation:** Store `was_new` on the `OAuthState` row when `set_result_token`
is called (pass it as a separate column or embed in the token), and return the
correct value from the exchange endpoint.

### I-4 — `.gitignore` missing common private-key file extensions

**File:** `.gitignore`
**Section:** §2

The `.gitignore` covers `.env`, `regstack.secrets.env`, and build artifacts, but
does not include `*.pem`, `*.key`, `*.p12`, or `*.pfx`. No such files exist in the
repository now, but the absence means a developer who generates a self-signed cert or
PEM key during testing and places it in the project root would not be warned before
committing it.

**Recommendation:** Add to `.gitignore`:
```
*.pem
*.key
*.p12
*.pfx
```

## 🟢 CLEAN (Passed review)

**§2 — Hardcoded secrets:** No API keys, private keys, connection strings with
credentials, or JWT tokens found in `src/`, `tests/`, `examples/`, or `docs/`.
Example `regstack.toml` files contain only placeholder values and no JWT secrets
(they rely on `REGSTACK_JWT_SECRET` env var). The file `src/regstack/config/secrets.py`
is correctly named — it contains key-derivation functions, not actual secrets.

**§3 — Authentication & authorization:** Every endpoint that returns or mutates
user state uses `rs.deps.current_user()` or `rs.deps.current_admin()`. All
responses use `UserPublic` projection (excludes `hashed_password`,
`tokens_invalidated_after`). Bulk-revoke discipline applied correctly: password
reset calls `update_password` (which bumps `tokens_invalidated_after`) + `lockout.clear`;
change-password does the same; confirm-email-change bumps the cutoff via
`update_email`. Admin disable (`is_active=False`) calls
`set_tokens_invalidated_after`. OAuth-only users (`hashed_password=None`) receive
the same generic 401 as a wrong-password attempt.

**§4 — Storage-layer injection:**
- *MongoDB:* All repos use structured PyMongo queries. No `$where`, `$expr`,
  `$function`, or `$accumulator` operators. No user-input regex patterns. Every
  `ObjectId` path parameter is guarded by `ObjectId.is_valid()` before construction.
  No unbounded `$in` arrays from user input.
- *SQL:* All queries use SQLAlchemy Core with parameterized binds throughout. The
  single `text("SELECT 1")` in `sql/backend.py` is a health-check ping with zero
  user input. No f-string or `%`-interpolated SQL found.

**§5 — Input validation:** All endpoints accept typed Pydantic models with
`extra="forbid"`. `EmailStr` on all email fields. E.164 format enforced via
`is_valid_e164()` before phone number storage. Jinja2 autoescape active on all
templates; no `| safe` filters or `{% autoescape false %}` blocks found in email
or SSR template trees. No shell-command injection surface in `regstack init` wizard.
Same-origin validation on `redirect_to` query parameters in OAuth start/link-start
endpoints (`_validate_redirect` correctly blocks `//evil.com` style bypasses).

**§6 — Rate limiting & DoS prevention:** Login lockout wired via `rs.lockout.check()`
before password verification in the login handler; `record_failure` fires for both
missing-user and wrong-password cases. Admin listing caps `limit` at 200 (`Query(ge=1,
le=200)`). OAuth state TTL enforced via Mongo TTL index and SQL `purge_expired()`.
No file-upload endpoints.

**§7 — Session & token security:** JWT algorithm is HS256 by default. Every token
includes `jti`. `iat` emitted as a float (RFC 7519 NumericDate). Bulk-revoke
comparison is `<=`. pyjwt's `exp`/`iat` checks disabled; both validated against the
injected `Clock`. `_authenticate` checks: decode → blacklist `jti` lookup →
`is_active` → bulk-revoke cutoff, in the correct order. No token values observed in
log calls.

**§8 — CSP and SSR layer:** No `<style>` blocks or `style="..."` attributes in any
SSR template. Single JS bundle loaded via `<script src defer>` — no inline
`<script>` tags anywhere. `oauth_complete.html` follows the same discipline. Pages
are stateless — no user email or auth-state fields rendered into HTML.

**§9 — Data exposure:** API responses use `UserPublic` consistently. Database driver
errors do not surface in HTTP responses. OAuth callback failures redirect with short
error codes, never exception text.

**§10 — Git history:** 30 most-recent commits reviewed. No security-sensitive
regressions detected. `git log --diff-filter=A` for key/env/secret filenames found
only the legitimate initial commit. No accidentally committed secrets, credentials,
or private keys.

**§11 — Third-party integrations:** SMTP password loaded as `SecretStr`, not echoed
in logs. AWS region/profile from config, not hardcoded. Twilio `auth_token` as
`SecretStr`. `google_client_secret` as `SecretStr`. Google ID-token verification
goes through `PyJWKClient` against the configured `jwks_url`. Lazy imports for
`aioboto3` and `twilio` confirmed — base install doesn't pull optional backends.

**§12 — GitHub Actions (partial):** No `pull_request_target` trigger. No
self-hosted runners. No `PYPI_API_TOKEN` secret — publish uses OIDC trusted
publishing (`id-token: write`). No `${{ github.event.* }}` interpolations inside
`run:` blocks (GITHUB_REF used via shell env expansion, not direct template
injection). Tag-push trigger for publish is `v*`. Version verification reads from
`pyproject.toml` via `tomllib`, not `import regstack`. (See W-1 and W-2 for
pinning/permissions gaps.)

**§13 — PyPI package integrity (partial):** Wheel built via `hatchling` with
`packages = ["src/regstack"]` — `tests/`, `docs/`, `tasks.py`, `examples/` are
excluded by default. `base-install-smoketest` job in CI catches unconditional
optional-extra imports at module top-level. `pywebview`, `aioboto3`, and `twilio`
are lazily imported. `pymongo`, `asyncpg`, `pyjwt[crypto]`, `aioboto3`, `twilio`
are all in optional extras, not base dependencies. (See W-6 for pywebview/uvicorn
exceptions.)

**§14 — OAuth subsystem:** PKCE `code_verifier` stays server-side (stored on state
row, never in the authorization URL). State rows consumed atomically
(`find_one_and_delete` on Mongo; SELECT+DELETE in one transaction on SQL). ID-token
validation checks signature (via `PyJWKClient`), `iss`, `aud`, `exp`, and `nonce`.
Auto-link only fires when `auto_link_verified_emails=True` **and**
`email_verified=True`. `identity_in_use` / `already_linked` / `email_in_use`
conflict codes properly differentiated. `DELETE /oauth/{provider}/link` refuses when
user has no password and no other identity.

**§15 — Multi-backend parity:** CI matrix covers Python 3.11 and 3.12 with MongoDB
7 and Postgres 16 service containers; SQLite via in-process aiosqlite. SQL migration
0002 uses `batch_alter_table` for SQLite compatibility. `UtcDateTime` TypeDecorator
correctly re-attaches UTC on reads from SQLite. Both SQL and Mongo backends enforce
`(provider, subject_id)` and `(user_id, provider)` unique constraints.

   `pypa/gh-action-pypi-publish@release/v1`, which currently uses a floating branch
