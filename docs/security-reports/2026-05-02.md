## 🟠 WARNING (Address within 1 week)

### W-1 · JWT decode error leaks internal reason to client

**File:** `src/regstack/auth/dependencies.py:120`

```python
raise HTTPException(
    status_code=status.HTTP_401_UNAUTHORIZED,
    detail=f"Invalid token: {exc}",   # ← exc message exposed
    headers={"WWW-Authenticate": "Bearer"},
) from exc
```

The raw `pyjwt` exception message (e.g. `"Signature verification failed"`,
`"The token is expired"`, `"Invalid header string"`) is returned verbatim to the
client in the 401 response body. The security review policy is explicit: *"JWT
decode errors → 401 generic, never the specific reason."*  
**Fix:** Replace with a static string such as `"Authentication required."` or
`"Invalid or expired token."`.

### W-2 · Admin `DELETE /admin/users/{id}` skips OAuth identity cascade

**File:** `src/regstack/routers/admin.py:131–134`

```python
await rs.users.delete(user.id)         # deletes the user
await rs.pending.delete_by_email(...)  # but NOT oauth_identities!
```

The user-initiated `DELETE /account` correctly cascades with:

```python
await rs.oauth_identities.delete_by_user_id(user.id)   # ← present
await rs.users.delete(user.id)
```

The admin delete path is missing this step. Any user with a linked Google
identity who is deleted by an admin will leave a dangling row in
`oauth_identities` that references a non-existent user ID. The CLAUDE.md
rule is: *"Account deletion must cascade `oauth_identities` first
(anti-orphan), then delete the user."*  
**Fix:** Add `await rs.oauth_identities.delete_by_user_id(user.id)` before
`await rs.users.delete(user.id)` in `admin.delete_user`.

### W-3 · `POST /phone/start` and `DELETE /phone` call `verify()` with potentially-`None` hash

**File:** `src/regstack/routers/phone.py:65, 149`

Both phone endpoints check the current password with:

```python
if not rs.password_hasher.verify(payload.current_password, user.hashed_password):
```

without first calling `_require_password_set(user)`. An OAuth-only user
(`hashed_password=None`) can hold a valid session JWT (issued by the OAuth
flow), reach these endpoints, and pass `None` as the stored hash to
`pwdlib`'s `verify`. The `PasswordHasher.verify` signature types `hashed: str`;
passing `None` will raise `TypeError` inside `pwdlib`, surfacing as an
unhandled 500 error instead of a clean 400.  
**Fix:** Add `_require_password_set(user)` as the first guard in both
`phone/start` and `phone/disable`, mirroring `change-password`,
`change-email`, and `delete-account`.

### W-4 · GitHub Actions: third-party actions pinned to tags, not commit SHAs

**Files:** `.github/workflows/publish.yml`, `.github/workflows/test.yml`

Every `uses:` directive references a mutable version tag:

| Action | Tag used |
|--------|----------|
| `actions/checkout` | `@v4` |
| `astral-sh/setup-uv` | `@v3` |
| `actions/upload-artifact` | `@v4` |
| `actions/download-artifact` | `@v4` |
| `pypa/gh-action-pypi-publish` | `@release/v1` |

Tag references can be silently rewritten by the action author (intentionally
or via a supply-chain compromise). `pypa/gh-action-pypi-publish` in particular
runs with the OIDC `id-token: write` permission and controls what lands on
PyPI — this is the highest-value target.  
**Fix:** Pin each action to its full commit SHA. Example:
`actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683 # v4.2.2`.
Use Dependabot's `actions` ecosystem or `pinact` to keep SHAs current.

### W-5 · GitHub Actions: missing `permissions:` blocks on most jobs

**Files:** `.github/workflows/publish.yml` (build job), `.github/workflows/test.yml` (all jobs)

Only the `publish` job in `publish.yml` declares explicit permissions
(`id-token: write`). Every other job — `build`, `pytest`, `docs`,
`base-install-smoketest` — runs without a `permissions:` block, meaning they
inherit the repository default. If the repository default is `write-all` (the
old GitHub default for public repos), these jobs receive more access than they
need.  
**Fix:** Add `permissions: { contents: read }` to each job that only needs
to check out and read code. The `publish` job already has the correct
minimal scope.

## 🟡 INFO (Best practice / hardening recommendations)

### I-1 · `OAuthConfig.completion_ttl_seconds` is declared but never consumed

**File:** `src/regstack/config/schema.py:71`

```python
completion_ttl_seconds: int = 30
```

This field exists in the config but is referenced nowhere in `routers/oauth.py`
or any repository. It was likely intended to limit how long the
`oauth_states.result_token` slot remains claimable after the callback completes
(before the SPA's `/oauth/exchange` call). Currently the full `state_ttl_seconds`
(300 s) applies, meaning an attacker who learns a `state_id` from logs could
attempt the exchange up to 5 minutes after the callback.  
**Action:** Either wire `completion_ttl_seconds` to update `expires_at` when
`set_result_token` is called, or remove the dead field.

### I-2 · `ExchangeResponse.was_new_account` is always `False`

**File:** `src/regstack/routers/oauth.py:153`

```python
return ExchangeResponse(
    ...
    was_new_account=False,   # ← hardcoded; _resolve_user returns was_new correctly
    ...
)
```

The internal flow correctly tracks `was_new` (via `_resolve_user`) and fires
hooks with it, but it is discarded before the SPA sees it. SPAs that want to
show a "Welcome, you just created an account!" flow cannot do so.  
**Action:** Pass `was_new` from `_resolve_user` (stored on the `OAuthState` row
or via a short-lived in-memory registry keyed on `state_id`) into the exchange
response.

### I-3 · `KNOWN_EVENTS` is incomplete

**File:** `src/regstack/hooks/events.py`

Events fired by routers but absent from `KNOWN_EVENTS`:

| Event fired | Where |

| `verification_requested` | `register.py`, `verify.py`, `admin.py` |

### I-4 · Login: no lockout failure recorded for disabled accounts

**File:** `src/regstack/routers/login.py:72–76`

```python
if not user.is_active:
    raise HTTPException(status_code=403, detail="Account is disabled.")
    # ← no rs.lockout.record_failure() call
```

An attacker who knows (or guesses) a disabled account's email address can send
unlimited login attempts against it without triggering lockout. Since the
account is disabled, the practical impact is limited to account-status
enumeration (403 vs 401). However, recording failures would eliminate even
that signal.  
**Action:** Call `await rs.lockout.record_failure(payload.email)` before the
403 raise, consistent with the treatment of non-existent accounts.

### I-5 · Google JWKS fetch has no explicit timeout

**File:** `src/regstack/oauth/providers/google.py:82`

```python
self._jwks_client = PyJWKClient(jwks_url, cache_keys=True)
```

`PyJWKClient` uses `urllib.request.urlopen` internally. Without an explicit
`timeout` parameter, a slow or unresponsive JWKS endpoint will block the OAuth
callback indefinitely (Python's default `urlopen` has no timeout). This can
freeze every OAuth login during a JWKS outage.  
**Action:** Upgrade to `PyJWKClient(jwks_url, cache_keys=True, timeout=5)` (or
an appropriate value). PyJWT ≥ 2.8 accepts this parameter.

### I-6 · `publish.yml` has a `workflow_dispatch` trigger without an explicit guard

**File:** `.github/workflows/publish.yml:12`

```yaml
on:
  push:
    tags: ["v*"]
  workflow_dispatch:
```

A manual trigger is useful for testing but allows any maintainer with Actions
write access to kick off a build without a tag. The `publish` job is gated on
`if: startsWith(github.ref, 'refs/tags/v')`, so an untagged dispatch won't
publish, but it does build and upload the wheel as an artifact. If the
`workflow_dispatch` trigger is not needed day-to-day, removing it reduces the
blast radius.

**§2 · Hardcoded secrets** — None found across `src/`, `tests/`, `examples/`,
`docs/`. Example TOML files (`examples/*/regstack.toml`) contain no `jwt_secret`
values, no real DB credentials, and use `console`/`null` backends. The git
history scan (`git ls-files | grep -iE '\.env|secret|credential'`) returns only
`src/regstack/config/secrets.py`, which is a utility module.

**§3 · Auth/authz** — All authenticated endpoints inject `current_user()` or
`current_admin()`. All state-mutation endpoints require authentication. All API
responses use `UserPublic` (excludes `hashed_password`, `tokens_invalidated_after`,
`updated_at`). Bulk-revoke discipline is complete for password reset, password
change, email-change confirm, and admin disable. Constant-time password
verification via `pwdlib/argon2`. JWT keys derived per-purpose via HMAC-SHA256.
OAuth-only users (`hashed_password=None`) correctly get the same generic 401 on
login (no enumeration); `change-password`/`change-email`/`delete-account` all
call `_require_password_set`.

**§4a · MongoDB injection** — No `$where`, `$expr`, `$function`, or
`$accumulator` operators anywhere in `src/regstack/backends/mongo/`. No
user-controlled regex patterns. All URL-derived user IDs are guarded by
`ObjectId.is_valid()` in `get_by_id` and `delete` before use; downstream
mutation methods are only reachable after a prior `get_by_id` succeeds.

**§4b · SQL injection** — All queries use SQLAlchemy Core with parameterised
binds. The one `text(...)` call (`backend.py:75`) is `text("SELECT 1")` — a
no-arg health check. No f-string or `%`-style interpolation into SQL. `UtcDateTime`
TypeDecorator correctly re-attaches `UTC` tzinfo on reads (preventing timezone
mismatch errors in bulk-revoke comparisons).

**§5 · Input validation** — All endpoints use Pydantic models with
`extra="forbid"`. `EmailStr` on all email-bearing fields. `is_valid_e164()`
called before phone number storage. Email Jinja2 templates use
`select_autoescape(["html"])`. SSR Jinja2 UI environment also uses autoescape.
No `| safe` filters or `{% autoescape false %}` blocks found in any template.
`regstack init` CLI wizard writes config files; no user input is executed as a
shell command.

**§6 · Rate limiting** — `LockoutService.check()` is the first call in
`POST /login`. `record_failure` is called before the password check for
user-not-found cases. Both Mongo (TTL index keyed on `login_lockout_window_seconds`)
and SQL (`purge_expired` + read-side `count_recent` with a parameterised cutoff)
backends are correctly wired. Admin `/users` listing caps `limit` at 200 via
`Query(ge=1, le=200)`. OAuth state rows have a 5-minute TTL (`state_ttl_seconds=300`)
enforced by Mongo TTL index and SQL `purge_expired`.

**§7 · Session & token security** — HS256, `jti` present in all tokens, `iat`
emitted as float (RFC 7519 NumericDate, not `int`) so microsecond-precision
bulk-revoke works correctly. `JwtCodec` disables pyjwt's `exp`/`iat` checks and
re-validates against the injected `Clock`. `_authenticate` checks: decode →
per-token blacklist → user active → bulk revoke — all four steps present in
correct order. No token values found in log calls.

**§8 · CSP / SSR layer** — Zero `<style>` blocks or `style="..."` attributes
in any template. Script loaded via `<script src defer>` only. `oauth_complete.html`
follows the same CSP discipline. SSR pages are stateless; no `{{ user.email }}`
or other auth data rendered into HTML.

**§9 · Data exposure** — All API responses go through `UserPublic` (excludes
`hashed_password`, `tokens_invalidated_after`). The sole f-string exception in
an HTTP response is in `_authenticate` (see W-1 above). DB driver errors are not
echoed to clients. OAuth callback errors redirect to `?error=<short_code>`,
never the raw exception. `client_secret` / SMTP password / Twilio auth token
all stored as `SecretStr`; SMTP password unwrapped only at `send()` time.

**§10 · Git history** — `git log -30` shows no security-sensitive commits in
the last 30 entries. `git log --diff-filter=A -- '*.env' '*.key' ...` returns
no results (no sensitive files ever added). The only `secrets`-named file in
git is `src/regstack/config/secrets.py` (utility module).

**§11 · Third-party integrations** — SMTP password via `SecretStr.get_secret_value()`.
AWS profile/region from config, no hardcoded values. Google `client_secret`
stored as `SecretStr`, unwrapped at provider construction only. `PyJWKClient`

**§12 · GitHub Actions** — All findings captured in W-4 and W-5 above. No

**No critical vulnerabilities found.** The codebase has solid fundamentals:
correct JWT revocation, per-purpose key derivation, Argon2id hashing,
anti-enumeration on sensitive flows, parameterised DB queries, and CSP-friendly
templates.

**Top 3 priorities for the development team:**

1. **W-1** — Patch the JWT error leak in `_authenticate` (one-line fix, high
   informational-disclosure impact).
2. **W-2** — Add OAuth identity cascade to admin user delete (straightforward
   one-line fix, prevents orphan rows that could cause integrity anomalies
   post-deletion).
3. **W-3** — Add `_require_password_set` guard to phone endpoints (prevents
   potential 500 errors for OAuth-only users, improves UX).

**Dependency health:** All locked versions are at or above the fix versions for
recent CVEs. No yanked packages detected.

**Actions workflow:** The biggest structural risk is un-pinned action tags
(W-4), particularly `pypa/gh-action-pypi-publish@release/v1` which can publish
to PyPI. SHA-pinning all actions is the highest-leverage hardening step available
in GitHub Actions.
