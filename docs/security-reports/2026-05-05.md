## 🔴 CRITICAL (Immediate action required)

### C1 — JWT decode error reason leaked to HTTP clients

**File:** `src/regstack/auth/dependencies.py`, line 121

```python
detail=f"Invalid token: {exc}",
```

`TokenError` is raised by `pyjwt.decode` and its message is the raw `pyjwt.PyJWTError` string (e.g. `"Signature verification failed"`, `"Signature has expired"`, `"Not enough segments"`). These strings are forwarded verbatim to the `WWW-Authenticate` challenge and JSON body that unauthenticated callers receive. Leaking the specific decode failure helps an attacker characterise the token format accepted by the server, confirm which algorithm is in use, and differentiate expired-but-valid-signature tokens from forgeries — all useful for replay or brute-force attacks.

**Fix:** Replace with a constant: `detail="Authentication required."` (matching the credential-absent branch).

## 🟠 WARNING (Address within 1 week)

### W1 — `pywebview` and `uvicorn[standard]` are in the base `dependencies` list

**File:** `pyproject.toml`, lines 37–39

```toml
"pywebview>=5.0",
"tomlkit>=0.13",
"uvicorn[standard]>=0.29",
```

These packages are required only by the `regstack oauth setup` and `regstack theme design` CLI wizards. Every downstream host that runs `pip install regstack` is forced to install `pywebview` (which transitively pulls in platform-specific GUI libraries: `pyobjc-*` on macOS, GTK/WebKit2GTK on Linux, webview2loader on Windows) and `uvicorn[standard]` (which pulls in `httptools`, `uvloop`, `websockets`). This significantly bloats the install surface of a library whose primary value is an embeddable FastAPI module. It also means GUI and web-server vulnerabilities become regstack's responsibility on every host, not just those that run the wizard.

**Fix:** Move `pywebview`, `uvicorn[standard]`, and `tomlkit` to a new `wizard` optional-extra (`[project.optional-dependencies]`). Add a guard in the wizard CLI entry points that raises a clear `ImportError` with an install hint if the extra is absent.

### W2 — `POST /phone/start` and `DELETE /phone` pass `user.hashed_password` (which may be `None`) to `PasswordHasher.verify`, causing an unhandled `TypeError` for OAuth-only users

**Files:** `src/regstack/routers/phone.py`, lines 65 and 149

```python
if not rs.password_hasher.verify(payload.current_password, user.hashed_password):
```

`user.hashed_password` is `str | None`. For OAuth-only accounts (created via Google sign-in) it is `None`. `PasswordHasher.verify` delegates to `pwdlib`, which raises `TypeError: hash must be str or bytes` when the second argument is `None`. FastAPI will convert the unhandled `TypeError` into an HTTP 500 Internal Server Error, leaking a stack trace to logs and returning an opaque error to the client.

The user-initiated account-management endpoints (`/change-password`, `/change-email`, `/account` DELETE) correctly call `_require_password_set()` before touching the password field. The phone router was not updated to match when OAuth support was added.

**Fix:** Add `_require_password_set(user)` (imported from `routers.account` or factored into a shared location) at the top of both `start` and `disable` before calling `password_hasher.verify`.

### W3 — `DELETE /admin/users/{id}` does not cascade OAuth identity rows

**File:** `src/regstack/routers/admin.py`, lines 131–134

```python
await rs.users.delete(user.id)
await rs.pending.delete_by_email(user.email)
```

The user-initiated `DELETE /account` flow (in `routers/account.py`, line 233) explicitly calls `await rs.oauth_identities.delete_by_user_id(user.id)` first. The admin delete path does not. There are no foreign-key constraints in either the SQL or MongoDB schemas that would cascade-delete identity rows automatically. After an admin hard-deletes a user, orphaned `oauth_identities` rows remain. If the same provider subject-id later signs up again, the unique index on `(provider, subject_id)` will reject the insert with `OAuthIdentityAlreadyLinkedError`, locking out the re-registering user permanently.

**Fix:** Add `await rs.oauth_identities.delete_by_user_id(user.id)` before `await rs.users.delete(user.id)` in `admin.py:delete_user`, matching the pattern in `account.py:delete_account`.

### W4 — GitHub Actions: no `permissions:` block on the `test` workflow or its jobs

**File:** `.github/workflows/test.yml`

The `test` workflow has no top-level or per-job `permissions:` key. With no explicit declaration the workflow inherits the organisation/repository default, which for most setups is `contents: read` + `pull-requests: read` — broader than the `{}` (no permissions) that a test-only workflow needs. If the repository's default is ever widened (e.g. to allow auto-labelling PRs), the test workflow would silently gain write access.

**Fix:** Add `permissions: {}` at the top level of `test.yml`. If the `upload-artifact` step requires any permission, scope it to `actions: write` on that job only.

### W5 — GitHub Actions: third-party actions pinned to mutable version tags, not commit SHAs

**Files:** `.github/workflows/test.yml`, `.github/workflows/publish.yml`

Every `uses:` directive references a floating tag:

```yaml
- uses: actions/checkout@v4
- uses: astral-sh/setup-uv@v3
- uses: actions/upload-artifact@v4
- uses: actions/download-artifact@v4
- uses: pypa/gh-action-pypi-publish@release/v1
```

Tags can be force-pushed by the action author at any time, silently replacing the code that runs in CI — this is a documented supply-chain attack vector. `pypa/gh-action-pypi-publish@release/v1` runs with `id-token: write` permission and is the direct path from the build job to PyPI; it is the highest-value target.

**Fix:** Pin every `uses:` to a full commit SHA. Use `actions/checkout@<sha>`, `astral-sh/setup-uv@<sha>`, etc. Tooling such as `pinact` or Dependabot's `update-type: github-actions` keeps SHAs current automatically.

### W6 — `pyjwt` lower bound (`>=2.8`) does not cover CVE-2026-32597

**File:** `pyproject.toml`, line 22

```toml
"pyjwt>=2.8",
```

CVE-2026-32597 (fixed in pyjwt 2.12.0, latest 2.12.1): PyJWT accepted JWS tokens that listed unknown `crit` (Critical Header Parameter) extensions rather than rejecting them as required by RFC 7515 §4.1.11. A host that uses regstack on a pyjwt version between 2.8 and 2.11.x is exposed to this vulnerability.

The `uv.lock` already pins pyjwt to `2.12.1`, so the current developer environment is safe. However, a host running `pip install regstack` without a lock file could resolve to any version ≥2.8 and land on a vulnerable pyjwt.

**Fix:** Raise the lower bound to `pyjwt>=2.12.1` in `pyproject.toml` (base `dependencies` and the `oauth` extra). Similarly consider raising the `jinja2` bound to `>=3.1.6` (CVE-2025-27516, sandbox breakout fixed in 3.1.6 — does not affect regstack's server-side rendering since templates are not attacker-controlled, but is good hygiene for hosts using the same environment).

## 🟡 INFO (Best practice recommendations)

### I1 — `KNOWN_EVENTS` in `hooks/events.py` is incomplete

**File:** `src/regstack/hooks/events.py`

Several events fired by the router are absent from `KNOWN_EVENTS`:

| Fired by | Event name | In KNOWN_EVENTS? |
|---|---|---|
| `routers/account.py` | `email_change_requested` | No |
| `routers/account.py` | `email_changed` | No |
| `routers/verify.py`, `admin.py`, `register.py` | `verification_requested` | No |
| `routers/phone.py` | `phone_setup_started` | No |
| `routers/phone.py` | `mfa_enabled` | No |
| `routers/phone.py` | `mfa_disabled` | No |
| `routers/login.py` | `mfa_login_started` | No |

`KNOWN_EVENTS` is the discoverable API contract for hosts wiring hooks. Missing events reduce the surface a host can subscribe to without reading source code. (`HookRegistry.on` does not validate against `KNOWN_EVENTS`, so this is not a security bug, but the incomplete list misleads host authors.)

**Fix:** Add the seven missing event names to `KNOWN_EVENTS`. Document each with a brief comment explaining what kwargs it carries.

### I2 — `oauth_signin_completed` in KNOWN_EVENTS is never fired at the right moment

**File:** `src/regstack/hooks/events.py` vs. `src/regstack/routers/oauth.py`, line 306

`oauth_signin_completed` is in `KNOWN_EVENTS` and is fired in the callback handler. However, it fires before the result token is stored on the state row — a handler that tries to read the user's session state may see an inconsistent view. This is low-severity because hooks are best-effort and handlers are not expected to rely on session state.

**Fix:** Move the `hooks.fire("oauth_signin_completed", ...)` call to after `oauth_states.set_result_token`.

### I3 — `POST /login/mfa-confirm` does not call `lockout.record_failure` on wrong codes

**File:** `src/regstack/routers/login.py`, lines 115–118

Wrong MFA codes at the second step are bounded by `mfa_codes.max_attempts` (default 5, via `config.sms_code_max_attempts`), but they do not increment the per-email lockout counter. An attacker who completes the first step (password correct, account not locked) can exhaust MFA codes without those failures contributing to the email-level lockout threshold. The `mfa_codes` table's own attempt tracking prevents brute-forcing individual code windows; this gap only matters if an attacker can cycle sessions quickly.

**Fix:** Call `await rs.lockout.record_failure(user.email)` in `mfa_confirm` when `result.outcome` is `MfaVerifyOutcome.WRONG`. Requires fetching the user before the verify call, or threading the email through the MFA token payload.

### I4 — `PyJWKClient` JWKS fetch has no explicit timeout

**File:** `src/regstack/oauth/providers/google.py`, line 82

```python
self._jwks_client = PyJWKClient(jwks_url, cache_keys=True)
```

`PyJWKClient` uses `urllib.request.urlopen` internally. The `PyJWKClient` constructor in pyjwt 2.x accepts a `timeout` parameter (added in 2.6.0). Without it, JWKS fetches use the Python default (no timeout), meaning a slow or stalled Google JWKS endpoint could hang a callback worker indefinitely.

**Fix:** Pass `timeout=10` to `PyJWKClient(jwks_url, cache_keys=True, timeout=10)`.

### I6 — `was_new_account` is always `False` in the `/oauth/exchange` response

**File:** `src/regstack/routers/oauth.py`, line 151–155

```python
return ExchangeResponse(
    access_token=state.result_token,
    redirect_to=state.redirect_to,
    was_new_account=False,   # hardcoded
    ...
)
```

The `ExchangeResponse.was_new_account` field is documented as indicating whether this was a new sign-up, but is hardcoded `False`. A host relying on this to trigger a "welcome" flow will never trigger it. This is not a security issue, but it is a documented API guarantee that is not honoured.

**Fix:** Store `was_new_account` on the `oauth_states` row during the callback (alongside `result_token`) and return it from `exchange`.

### I7 — `verification_requested` hook passes the raw verification URL (and thus token) to handlers

**Files:** `src/regstack/routers/register.py` line 97, `src/regstack/routers/verify.py` line 101, `src/regstack/routers/admin.py` line 171

```python
await rs.hooks.fire("verification_requested", email=payload.email, url=url)
```

The `url` parameter contains the raw, single-use verification token in the query string. Hook handlers that log or forward this URL (e.g. to an analytics system) effectively publish a usable token. This is intentional for hosts that need the URL for custom email delivery, but is worth calling out in the hook documentation so host authors don't accidentally log it.

**Fix (documentation):** Add a docstring or comment to `KNOWN_EVENTS` for `verification_requested` and `email_change_requested` noting that the `url` kwarg contains a security credential and must not be logged.

Same pattern as I7 but for the email-change token. The `url` passed to the hook contains a short-lived JWT that can be used to change the user's email address without re-authentication. Same documentation recommendation applies.

**Section 2 (Hardcoded Secrets):** No API keys, private keys, JWTs, or production credentials found in `src/`, `tests/`, or `examples/`. The only flagged file in `git ls-files` matching `*secret*` is `src/regstack/config/secrets.py`, which contains the derivation function, not a secret value. Example `regstack.toml` files use placeholder emails and localhost URLs only.

**Section 3 (Authentication/Authorisation — core):** All authenticated endpoints correctly inject `rs.deps.current_user()` or `rs.deps.current_admin()`. The bulk-revoke chain is complete: `update_password`, `update_email`, and `set_tokens_invalidated_after` all bump `tokens_invalidated_after`. Admin disable correctly calls `set_tokens_invalidated_after`. OAuth-only users receive a generic 401 from `POST /login` — no enumeration. `UserPublic` is used consistently for all API responses; `hashed_password` and `tokens_invalidated_after` never appear in response bodies. Constant-time password verification via `pwdlib` (not `==`).

**Section 4 (Storage Injection):** MongoDB repos use parameterised driver calls exclusively. No `$where`, `$expr`, or regex built from user input. `ObjectId.is_valid()` guards all `ObjectId(user_id)` calls. SQL repos use SQLAlchemy Core with bound parameters throughout; no `text()` calls with interpolation. `UtcDateTime` TypeDecorator correctly re-attaches UTC on read for both SQLite and Postgres.

**Section 5 (Input Validation):** All route handlers accept Pydantic models with `extra="forbid"`. `EmailStr` used on all email fields. `is_valid_e164()` called on all phone number inputs. Jinja2 autoescape is on for all HTML templates (no `| safe` filters or `{% autoescape false %}` blocks found). SSR templates contain no inline `<style>` blocks or `style="..."` attributes. `redirect_to` in OAuth flows is validated against `config.base_url` via `_validate_redirect`.

**Section 6 (Rate Limiting):** `LockoutService.check` is called before password verification in the login route — a locked-out attacker cannot probe password correctness. `login_attempts` Mongo TTL index is sized to `login_lockout_window_seconds`. SQL backends use `purge_expired()` read-side filtering. Admin `/users` list caps `limit` at 200 (`Query(ge=1, le=200)`). OAuth state TTL is enforced. No file-upload endpoints exist.

**Section 7 (Session/Token Security):** `JwtCodec` issues `iat` as a float (`self.iat.timestamp()` in `to_claims()`). PyJWT `exp`/`iat` checks are disabled and re-implemented against the injected `Clock`. Every token includes a `jti`. `is_payload_bulk_revoked` uses `<=` comparison. `_authenticate` performs the full check chain: decode → blacklist → user active → bulk revoke.

**Section 8 (CSP/SSR):** No `<style>` blocks, `style="..."` attributes, or inline `<script>` tags found in any template. `regstack.js` loaded via `<script src defer>`. `oauth_complete.html` follows the same discipline. SSR pages are stateless — no `{{ user.email }}` or similar rendered into HTML.

**Section 9 (Data Exposure — partial; see C1 for token detail leak):** Admin and account endpoints use `UserPublic.from_user()`. JWT decode errors in `reset-password`, `confirm-email-change`, `phone/*`, and `login/mfa-confirm` return safe, generic messages. OAuth callback failures redirect with a short code, never the exception text.

**Section 10 (Git History):** 30 recent commits reviewed. The commit `5ad8847` (initial) added `src/regstack/config/secrets.py` (the derivation function, not a secret). No sensitive files (`.env`, `.key`, `.pem`) detected in any commit via `git log --diff-filter=A`.

**Section 11 (Third-party Integration):** `EmailConfig.smtp_password` is `SecretStr`. `OAuthConfig.google_client_secret` is `SecretStr`. SMTP password extracted via `.get_secret_value()` and never logged. AWS credentials come from the environment/profile (no hardcoded values). Twilio `auth_token` is `SecretStr`. Token exchange HTTP client has a 10-second timeout.

**Section 12 (GitHub Actions — partial; see W4 and W5):** No `pull_request_target` trigger. No self-hosted runners. No `secrets.*` usage in `run:` scripts — the publish workflow uses OIDC trusted publishing with no PyPI token. No `${{ github.event.* }}` interpolations in shell commands.
